# CV Preview srcDoc Migration — Design Spec

**Date:** 2026-04-02  
**Sprint:** 8 (Iteration 22)  
**Status:** Approved  

---

## Problem

The CV preview iframe is blocked by Firefox (and any browser enforcing strict CSP) because the Next.js frontend (`localhost:3000`) and FastAPI backend (`localhost:8001`) are different origins. The backend's `GET /api/cv/{id}/html` endpoint returns:

```
X-Frame-Options: SAMEORIGIN
Content-Security-Policy: frame-ancestors 'self'
```

These headers are correct for a same-origin deployment but break local and cross-origin dev setups. Chrome silently allowed it in some configurations, masking the bug.

---

## Decision

Replace `<iframe src={url}>` with a fetch-and-inject `<iframe srcDoc={html}>` pattern.

**Why this works:**
- `srcDoc` iframes have an `about:srcdoc` origin, treated as same-origin to the parent
- `X-Frame-Options` and `frame-ancestors` are not evaluated for `srcDoc` content — only for navigated (`src=`) frames
- The Jinja2 CV templates use fully inline CSS — no external asset references — so the HTML is self-contained
- Style isolation between the CV and the Next.js app is preserved by the iframe boundary

**Security:** The HTML is generated by backend-controlled Jinja2 templates, not user-supplied content. The iframe uses `sandbox="allow-same-origin"` for defense-in-depth (blocks scripts, forms, popups inside the preview).

## Architecture

No structural changes to the system. The API contract, Jinja2 rendering pipeline, and PDF generation path are all unchanged. This is a rendering technique change within `CVPreview.tsx`.

```
Frontend (fetch)  →  GET /api/cv/{id}/html  →  Backend (returns HTML string)
                  ←  HTML string             ←
Frontend injects html into <iframe srcDoc={html}>
```

## Component Design — CVPreview.tsx

### State

```typescript
const [htmlContent, setHtmlContent] = useState<string | null>(null);
const [previewError, setPreviewError] = useState(false);
const [retryCount, setRetryCount] = useState(0);
```

### Effect

Runs on `[cvId, retryCount]`. Resets both `htmlContent` and `previewError` at the start of each run. Uses a `cancelled` flag for cleanup to prevent stale state updates when `cvId` changes mid-fetch.

```typescript
useEffect(() => {
  let cancelled = false;
  setHtmlContent(null);
  setPreviewError(false);

  fetch(`${API_BASE}/api/cv/${cvId}/html`)
    .then((res) => {
      if (!res.ok) throw new Error("Failed to load preview");
      return res.text();
    })
    .then((html) => { if (!cancelled) setHtmlContent(html); })
    .catch(() => { if (!cancelled) setPreviewError(true); });

  return () => { cancelled = true; };
}, [cvId, retryCount]);
```

### Render States

| State | Condition | UI |
|-------|-----------|-----|
| Loading | `!htmlContent && !previewError` | `div` with `animate-pulse bg-gray-100` skeleton |
| Loaded | `htmlContent !== null` | `<iframe srcDoc={htmlContent} sandbox="allow-same-origin" data-testid="cv-iframe">` |
| Error | `previewError === true` | Error message + "Erneut versuchen" button |

### Retry

```typescript
onClick={() => { setPreviewError(false); setRetryCount(c => c + 1); }}
```

`retryCount` is in the effect's dependency array, so incrementing it re-triggers the fetch without requiring a `cvId` change.

### Props Interface

Unchanged — `CVPreviewProps` is not modified.

## Backend — cv.py

Remove the headers dict from `get_html()`:

```python
# FROM:
return HTMLResponse(
    content=html,
    headers={
        "X-Frame-Options": "SAMEORIGIN",
        "Content-Security-Policy": "frame-ancestors 'self'",
    },
)

# TO:
return HTMLResponse(content=html)
```

Auth (`get_auth_provider`) and CORS remain. The endpoint is still protected.

## Testing

### Unit Tests (Vitest + RTL) — Task 22.5

**Setup:** Add to `frontend/package.json` devDependencies:
- `vitest`, `jsdom`, `@vitejs/plugin-react`
- `@testing-library/react`, `@testing-library/user-event`

**Config:** `frontend/vitest.config.ts` with `environment: 'jsdom'` and `globals: true`.

**Test file:** `frontend/components/cv/__tests__/CVPreview.test.tsx`

Four test cases, all mocking `fetch`:
1. **Loading state** — `fetch` is a never-resolving promise; assert skeleton `div` is rendered, iframe is absent
2. **Loaded state** — `fetch` resolves with HTML string; assert `[data-testid="cv-iframe"]` has correct `srcdoc`
3. **Error state** — `fetch` rejects; assert error message and retry button are visible
4. **Retry** — after error state, click retry button; assert fetch is called a second time

### E2E Tests (Playwright) — Tasks 22.2 & 22.4

**Config change:** Uncomment Firefox project in `playwright.config.ts`.

**New spec:** `tests/e2e/cv-preview.spec.ts`
1. Navigate to CV preview screen
2. Assert `[data-testid="cv-iframe"]` is present
3. Assert `srcdoc` attribute is non-empty
4. Assert visible text content inside the iframe (candidate name from Jinja2 template)
5. Assert no console errors related to frame blocking or CSP violations
6. Smoke test: click download button, assert download response (Content-Disposition header)

Run on both `chromium` and `firefox` projects.

## Acceptance Criteria

1. Firefox renders the CV preview without console errors or blank iframe
2. Chrome continues to render correctly (no regression)
3. `GET /api/cv/{id}/html` no longer returns `X-Frame-Options` or `frame-ancestors` headers
4. The iframe has `sandbox="allow-same-origin"` and `data-testid="cv-iframe"`
5. Loading skeleton visible during fetch
6. Network error shows German error message with retry button; retry re-fetches
7. Changing `cvId` (via regeneration) re-fetches and re-renders
8. PDF download continues to work in both browsers
9. All Playwright E2E tests pass on Chromium and Firefox
10. All Vitest unit tests pass
